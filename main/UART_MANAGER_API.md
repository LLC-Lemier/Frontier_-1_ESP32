# Контракт API `uart_manager` для внешних задач

Исходное ТЗ на Frontier: **`ТЗ/V1211_260409_150625.pdf`** от корня репозитория (см. **`HARDWARE.md`**). Для этапа **Frontier_1** (один UART, без SPI/XR20) см. также **`ТЗ/Frontier_1_requirements.txt`**. Ниже — соглашения по API в коде, без дублирования полного ТЗ.

Документ фиксирует правила вызова и синхронизации между задачей драйвера `uart_manager` и остальными задачами прошивки (мост TCP/UDP, обработчики HTTP, SNMP, диагностика).

На этапе **Frontier_1** под RX/TX кольца единственного порта при инициализации модуля резервируется память **дважды по 64 КиБ**; поля `ring_rx_size` / `ring_tx_size` в конфигурации задают **фактическую глубину** (не больше пула), без перевыделения при смене настроек — см. **`ТЗ/Frontier_1_requirements.txt`** (п. 3.4).

## Границы ответственности

| Слой | Действия |
|------|----------|
| Внутри `uart_manager` | SPI к XR20M1280, GPIO мультиплексора (IF_SEL_A/B, CS1…CS4), опрос/запись регистров UART, программные кольцевые буферы |
| Внешние задачи | Только публичные функции из `uart_manager.h` и при необходимости очереди/события, полученные при инициализации |

Прямой доступ к `driver/spi_master.h` и GPIO линий порта из прикладных задач **запрещён**.

## Контексты вызова

| Контекст | Правило |
|----------|---------|
| Задача FreeRTOS | Основной контекст для `uart_manager_send`, `uart_manager_receive`, `uart_manager_configure_port`, `uart_manager_get_stats` |
| Колбэки HTTP / lwip | Не вызывать блокирующие функции без таймаута; предпочтительно передать запрос в очередь и вернуть ответ асинхронно |
| ISR | Публичный API **не** вызывать, кроме функций с суффиксом `*_from_isr`, если такие появят в будущем |

## Синхронизация

- **Команды и данные**: внешняя задача постит байты на передачу через `uart_manager_send(..., timeout_ticks)`; читает из RX через `uart_manager_receive`. Внутри драйвера данные согласуются с кольцевыми буферами и задачей `uart_mgr`.
- **Задача `uart_mgr`**: снимает байты из RX-буфера ESP-IDF UART (`uart_read_bytes`) после ожидания по **двум** источникам — `xTaskNotify` (TX / остановка) и **очередь событий драйвера** (`UART_DATA`, переполнения и т.д.), в пределах окна ~20 ms для периодического статуса. Длительность одного блокирующего ожидания события задаётся Kconfig **`UART_MGR_RX_EVENT_WAIT_MS`** (меньше — быстрее реакция на notify при тишине на RX).
- **Статистика** (`uart_manager_get_stats`): счётчики обновляются под тем же мьютексом, что и доступ к кольцам — снимок консистентен.
- **События**: `uart_manager_wait_event` с таймаутом — готовность данных, **фронт** переполнения (`UART_MGR_EVT_RX_OVERFLOW` при новом инциденте, а не «пока счётчик > 0»), ошибка линии.
- **Конфигурация**: `uart_manager_configure_port` держит мьютекс на время обновления `cfg`, колец и **`uart_param_config`**, чтобы не было рассинхрона памяти и железа. Повторные вызовы из разных задач координируйте снаружи (внутри модуля только взаимное исключение, не очередь команд).

## Порты (Frontier_1)

Используйте индекс **`UART_MGR_F1_PORT_ID` (0)**. Для других значений `port`: функции с `esp_err_t` → **`ESP_ERR_NOT_SUPPORTED`** (однократное предупреждение в лог); `send` / `receive` / `wait_event` → **0** без передачи данных.

## Остановка модуля

`uart_manager_deinit` останавливает задачу `uart_mgr` корректно (ожидание до 5 с) и затем удаляет драйвер и пулы. Избегайте вызова deinit в ситуации, когда другая задача может долго удерживать API под мьютексом модуля.

## NVS

Чтение: допустимы блоб **28 B** (legacy v1) и **32 B** (v2 с CRC32). Неверный размер или CRC — конфиг не применяется, остаются дефолты. Запись всегда в формате v2 с CRC.

## Отладка

Опции Kconfig: **`UART_MGR_VERBOSE_LOG`** — подробные `ESP_LOGD` в задаче `uart_mgr`; **`UART_MGR_RX_EVENT_WAIT_MS`** — см. синхронизацию выше.

## Порядок инициализации

1. `network_config_init()` (инициализация NVS и сетевых настроек в прошивке)
2. `uart_manager_init()` — не зависит от Ethernet/EAP-TLS; локальные COM доступны до сети; внутри или сразу после — загрузка конфигурации порта из NVS (`uart_manager_load_config_from_nvs`), если это согласовано с реализацией `main` (см. **`ТЗ/Frontier_1_requirements.txt`**, п. 3.8: NVS для `uart_manager` **обязателен**, не заглушка).

Сохранение: `uart_manager_save_config_to_nvs()` вызывается после успешного изменения конфигурации (или по команде сети), когда это предусмотрено приложением.

## Потребители (план)

| Потребитель | Тип задачи | API |
|-------------|------------|-----|
| `app_main` | старт | `uart_manager_init`, опционально тестовая задача |
| Будущий TCP-мост | долгоживущая | `send` / `receive` / `wait_rx` |
| `web_api` | HTTP worker | только неблокирующие вызовы с таймаутом или отложенная обработка |

## Версия

Согласовано с этапом 1: функции и семантика таймаутов могут быть расширены без изменения префикса `uart_manager_`.
